# 各类业务在 Starlink 上网时的速率保障需求总结

本文总结了在使用 Starlink（或其他 LEO
卫星宽带）上网时，不同业务类型为了获得"良好体验"所需的
**可持续保障带宽（sustained throughput）**。并给出必要的工程级注意事项。

------------------------------------------------------------------------

## 📌 2. 工程注意事项（比带宽更重要）

### ① Starlink 的延迟与抖动

-   日常延迟 20--50 ms\
-   游戏/实时业务更看重延迟、抖动与丢包\
-   不稳定时需要额外 bandwidth headroom

### ② 上行带宽可能成为瓶颈

-   视频会议、直播、游戏语音依赖上行\
-   多用户同时上传可能触发拥塞

### ③ 建议预留裕度（Headroom）

实际部署中建议将业务需求 × **1.2--1.5**，用于应对：\
- 链路波动\
- 协议开销（TCP/IP、RTP/UDP）\
- 多用户并发\
- 卫星切换瞬态（handover 期间的 jitter）

------------------------------------------------------------------------

## 📌 3. 常用场景的速率配置模板

### ✔ 单用户观看 1080p 视频

-   **下行：8 Mbps**（含稳定裕度）\
-   上行需求极低

### ✔ 1080p 视频会议（双向）

-   **上行：≥4 Mbps**\
-   **下行：≥4 Mbps**\
-   严格要求：低抖动、低丢包

### ✔ 高质量语音（VoIP）

-   **24--64 kbps 上/下行**\
-   延迟敏感

### ✔ 在线游戏

-   **下行：3--10 Mbps**\
-   **上行：1--3 Mbps**\
-   关键是 **RTT \< 50--80 ms** + 稳定性

### ✔ 云游戏

-   **10--35 Mbps 下行**（取决画质）\
-   要求同样是低延迟稳定链路

------------------------------------------------------------------------

## 📌 4. 速率使用建议

1.  对每个流量类型设定最小保障带宽。\
2.  并发用户时，把所有业务速率相加，然后 ×1.3--1.5。\
3.  对实时业务（语音、会议、游戏）优先保障低延迟、低抖动。\
4.  为卫星切换（LEO handover）预留额外余量以避免中断。

------------------------------------------------------------------------

## 📌 5. 一句话总结

> Starlink
> 具有足够带宽支撑高清流媒体、游戏与会议，但要保障体验，需要对不同业务分配对应的最小带宽，并为卫星切换与链路波动预留裕度。
